Ultra-Fast Energy-Above-Hull (Ehull)
for High-Entropy DRX Cathodes

A vectorised, LP-batched, MPI-parallel algorithm that replaces PyMatGen's per-entry PhaseDiagram with system-level batch LP solving — enabling ~2 million Ehull calculations at a fraction of the cost.

⚡ PyMatGen vs. This Implementation

❌ PyMatGen PhaseDiagram (Standard)

  • One full PhaseDiagram object per query O(n) overhead
  • Per-entry Python loop — no vectorisation
  • QHull convex hull rebuilt each call
  • No batching: Pool overhead = 1 task/entry
  • Repeated composition parsing every entry
  • CSV output — slow, large files
  • No chemical-system caching
  • Practical limit ≈ tens of thousands entries

✅ This Implementation (LP-Batch MPI)

  • LP matrices built once per chemical system O(1) rebuild
  • NumPy matrix multiply for all formation energies
  • HiGHS C++ LP solver — not QHull convex hull
  • Pool task = whole chemical system (100× less overhead)
  • Composition matrix (X) computed once and reused
  • Parquet output — 10× smaller, 10× faster I/O
  • LRU-style LP cache pickled and resumed
  • Scales to ≥ 2 million entries on 16 CPUs
🔬 Algorithm Pipeline — Step by Step
1

Load & Process Reference Database

All Materials Project reference phases for the 19-element HE-DRX space (Li-Al-Co-Cr-F-Fe-Ga-Mg-Mn-Mo-Nb-Ni-O-Sb-Sn-Ti-V-Zn-Zr) are loaded from a local pickle and corrected with MaterialsProject2020Compatibility. Elemental chemical potentials μ[El] are extracted once — these serve as the formation-energy reference for every subsequent calculation.

🔑 Key advantageDone once. All 2 M targets share the same reference database and μ values — zero redundant API calls.
2

Group Targets by Chemical System

Each HE-DRX cathode is assigned a chemical-system key (alphabetically sorted element set, e.g. Co-Li-Mn-Ni-O-Ti). Entries are grouped into system buckets. Only systems with at least one reference phase proceed. Duplicate stoichiometries are deduplicated — only the lowest-FE representative per reduced formula is kept.

🔑 Key advantagePool tasks = # unique systems (hundreds), not # entries (millions). Serialisation overhead drops by ~100×.
3

Pre-compute LP Matrices per System

For each unique chemical system the reference entries with elements ⊆ system are assembled. A single NumPy matrix multiply (fe = epa − X @ μ_vec) computes all formation energies at once (BLAS level-2). The composition matrix A (k×n_refs) and cost vector c (n_refs,) are stored in the LP cache — built only once per system, reused for every target in that system.

🔑 Key advantageOne matrix multiply replaces millions of Python for-loops. Composition vectors are never re-parsed per entry.
4

Parallel System-Batch LP Solving (HiGHS)

Each worker receives one system's LP cache (A, c, μ_vec) plus all target entries for that system. The equality constraint matrix A_eq = [A; 1ᵀ] is assembled once; only the RHS b_eq = [x_i; 1] changes per target. HiGHS C++ LP solver (via scipy.linprog) is called per entry — much faster than QHull for high-dimensional simplex problems. Ehull = FE_target − LP_optimal.

🔑 Key advantageHiGHS solves each LP in microseconds. Shared A matrix means no repeated memory allocation per entry. 16-CPU parallelism across systems.
5

Optional: Voltage via PatchedPhaseDiagram

Decoupled from the Ehull path (flag COMPUTE_VOLTAGE). When enabled, a PatchedPhaseDiagram is built once (or loaded from cache) and Li chemical-potential profiles are computed per entry in a separate parallel pool. Voltage range [min_V, max_V] is extracted from transition chemical potentials.

🔑 Key advantageVoltage pass does not block Ehull — most runs skip it entirely, saving the heavy PPD construction cost.
6

Incremental Parquet Output & Resume

Results are written to Parquet (columnar binary format) — 10× smaller and faster than CSV for 2 M rows. On restart the existing Parquet is read to extract already-computed rel_dir keys; those entries are skipped. New results are appended and the file is overwritten atomically.

🔑 Key advantageCrash-safe resume: only uncomputed entries are processed. Parquet I/O is orders of magnitude faster than CSV for large datasets.
💡 Eight Core Innovations
🧮
Vectorised Formation Energies

All FE computed as a single BLAS matrix multiply epa − X@μ — replaces Python loops entirely.

📦
System-Level Batching

Pool task = 1 chemical system (hundreds of tasks) not 1 entry (millions). Cuts serialisation overhead 100×.

⚡
HiGHS LP Solver

State-of-the-art C++ LP solver instead of QHull. Handles high-dimensional simplex in microseconds per query.

♻️
Shared LP Matrix Cache

A and c built once per system, reused for every target. Only b_eq (the RHS) changes per entry.

🔀
16-CPU MPI Parallelism

Systems dispatched across 16 workers via Pool.imap_unordered with chunksize=1 for load balance.

🗂️
Parquet I/O

Columnar binary output is 10× smaller and faster to read/write than CSV for millions of rows.

🔁
Crash-Safe Resume

Already-computed entries detected from existing Parquet and skipped. No wasted computation on restart.

⚙️
Decoupled Voltage Pass

Voltage computation is isolated behind a flag — Ehull path runs at full speed without PPD overhead.
